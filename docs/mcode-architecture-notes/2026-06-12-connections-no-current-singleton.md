# Connections No Current Singleton

## Summary

`mcode-app` 不再把“当前连接”作为用户可见的单例状态。连接页允许多个连接并存在线；会话新建、待办发送等入口改为显式选择连接组，或沿用当前页面上下文，不再默认绑定到某个全局“当前连接”。

## Architecture

- 连接页只维护两个本地维度：
  - `connectedMap`：用户是否已建立该连接。
  - `onlineMap`：该连接当前是否探活在线。
- `auth` store 继续保留，但职责收敛为“单次网关调用前的运行时上下文注入”。
- `auth` 不再代表产品层的“当前连接”，也不驱动连接页的状态展示、切换确认或断开副作用。

## Data Flow

1. 用户在连接页点击“连接”时，仅对目标连接做 pair / session 恢复。
2. 成功后写入 `connectedMap`，并启动该连接自己的在线探活与重连 watcher。
3. 用户在会话新建、待办发送等场景中显式选择连接组。
4. 真正发起网关调用前，页面把所选连接同步到 `auth`，仅用于当前调用拿到正确的 gateway / remote instance descriptor。
5. 断开或删除某个连接时，只清理该连接本地 linked/online 状态，不清空其它并存连接。

## UI Behavior

- 连接卡片点击不再弹“切换连接”确认。
- 已在线连接再次点击或再次点菜单“连接”时，仅提示该连接已在线。
- 待办发送弹层默认选中第一个已连接分组。
- 会话历史页的新建入口直接在当前历史分组下可用，不再要求它等于某个全局当前连接。

## Compatibility

- 不修改后端协议。
- 现有 `auth` store、gateway、realtime instance key 机制继续可用。
- 旧代码若仍把 `auth.mode + url` 当成“当前连接”真值，会与新模型冲突，后续新增页面应避免复用该假设。

## Native Replication Guidance

- 原生端实现时，不要维护用户可见的“唯一当前连接”。
- 连接列表应支持多个连接同时处于“已连接/在线”状态。
- 所有需要目标连接的动作都应：
  - 从显式 UI 选择或当前页面上下文得到连接 key；
  - 再为本次调用构造 transport/gateway 上下文。
- 断开单个连接时，不要顺带清空其它连接或把整个应用重置为“未连接”。

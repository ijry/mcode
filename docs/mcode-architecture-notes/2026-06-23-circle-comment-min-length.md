# 2026-06-23 circle comment min length

## Architecture

圈子评论的最短长度校验必须前后端保持一致。前端负责即时交互提示，后端负责最终数据校验，任何一侧仍保留旧阈值都会导致“按钮可点但提交失败”或“点击无反应”的错觉。

## Protocol And Data Flow

- 前端圈子详情页与圈子列表评论面板统一使用 `2` 个字符作为最小评论长度
- 提交前先做本地校验，不满足时直接 `toast`
- 后端公共评论接口 `/v1/comment/comment/add` 同步校验为 `length:2,300`

这样数据流为：

1. 用户输入评论
2. 前端先判断 `trim().length >= 2`
3. 不满足时直接提示“评论至少 2 个字”
4. 满足后再调用后端提交
5. 后端用同样的 `2-300` 规则兜底校验

## UI Behavior

- 评论少于 2 个字时，不再静默无反应
- 发送按钮激活态与实际可提交条件一致
- 输入框占位文案同步改为“至少 2 个字”

## Compatibility

- 仅调整评论最短长度，不影响评论树、分页、排序与回复关系
- 兼容现有最大长度 `300`
- 若其他客户端复用同一评论接口，也必须同步最短长度为 `2`

## Native Replication

iOS / Android 原生端应复用相同规则：

- 本地输入校验：`trim().length >= 2`
- 不满足时即时提示，不要静默返回
- 服务端仍保留 `2-300` 兜底校验，避免绕过客户端提交非法内容

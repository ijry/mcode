# Circle Post Menu Edit Share Design

## Scope

This change adds two user-facing capabilities around circle posts:

1. Post action menus with `分享`, plus `编辑` for the author only.
2. Author-safe post editing backed by xycloud server-side ownership checks.

It also synchronizes the `up-markdown` `linktap` and `copyLink` passthrough
change into the source `uview-plus` repository so the patched behavior does not
live only inside `mcode-app/node_modules`.

## Goals

- Let users share a post from both feed cards and the post detail page.
- Let only the post author edit a post.
- Enforce edit ownership on the backend, not only in frontend UI.
- Reuse the existing publish page for edit mode instead of creating a second
  editor screen.
- Keep the external-link interception work compatible with upstream
  `uview-plus`.

## Non-Goals

- No delete-post capability in this change.
- No system-native share sheet integration in this change.
- No redesign of circle feed, detail layout, or comment behavior.

## Approach Options

### Option 1: Recommended

Use a shared frontend post-action menu helper, reuse the publish page in
`create/edit` modes, and add a mobile-facing xycloud edit endpoint with author
ownership validation.

Why this is recommended:

- Authorization is correct because the backend owns the final decision.
- The publish page already has topic/image/markdown handling and is the lowest
  risk place to reuse.
- Share behavior stays deterministic across H5 and app runtimes because it only
  writes to clipboard.

### Option 2

Add frontend menus and attempt to call an existing server edit path without
backend changes.

Why not recommended:

- It assumes the current circle controller already exposes a mobile-safe edit
  contract, which is not guaranteed.
- Ownership would still need server confirmation, so frontend-only work is not
  enough.

### Option 3

Split into two releases: share now, edit later.

Why not recommended:

- Leaves the menu contract half-finished.
- Duplicates future UI work because menu structure would change twice.

## Architecture

### Frontend

The frontend gets a small circle post action layer used by both:

- `src/pages/circles/index.vue`
- `src/pages/circles/detail.vue`

This layer decides which menu items to show for a given `CirclePost`:

- Always show `分享`
- Show `编辑` only when the post belongs to the current logged-in user

The existing publish page becomes dual-mode:

- Create mode: current behavior
- Edit mode: receives `id` in route params, loads the post detail, pre-fills
  the form, and submits to the edit API

Clipboard share content is generated by a shared helper so feed and detail page
produce the same text.

### Backend

`xycloud` circle post controller gets a mobile-facing edit action under the
front-end API namespace. The edit action accepts a post id plus editable fields,
loads the target post, checks the current login user, rejects non-authors, and
then persists the changes through the existing circle model/service flow.

The backend is the authority for:

- Login requirement
- Post existence
- Author ownership
- Field normalization and persistence

## Data Flow

### Share

1. User taps the three-dot menu in feed or detail.
2. User chooses `分享`.
3. Frontend builds:
   - title if present, otherwise content excerpt
   - mcode detail route link for the post
4. Frontend copies `<summary>\n<link>` to clipboard.
5. Frontend shows success or failure toast.

### Edit

1. User taps the three-dot menu.
2. If the post belongs to the current user, menu shows `编辑`.
3. User enters publish page in edit mode, e.g. `/pages/circles/publish?id=123`.
4. Publish page calls `fetchCirclePost(123)` and populates title, content,
   topic ids, and image list.
5. User modifies content and submits.
6. Frontend calls new service method for edit.
7. Backend validates current user owns the post.
8. Backend updates the post and returns success.
9. Frontend sets the existing refresh flag and navigates back.
10. Feed/detail reload shows updated content.

## API Contract

### Frontend Service Additions

Add a new method in `src/services/circle.ts`:

- `updateCirclePost(payload)`

Suggested payload:

- `id: number`
- `title: string`
- `content: string`
- `topicIds: number[]`
- `images: string[]`

### Backend Route

Expose a front-end route in xycloud circle post controller:

- `POST /v1/circle/post/edit/<id>`

Expected request fields:

- `title`
- `content`
- `topicIds`
- `images`

Expected validation:

- user must be logged in
- target post must exist
- target post `uid` must equal current user id

Failure responses should follow existing xycloud error style so current frontend
error normalization still works.

## UI Behavior

### Feed Card

The right-side `more-dot-fill` icon becomes clickable and opens an action sheet.
Its click must not trigger `openPostDetail(post)`.

### Detail Page

The top-right share button stays available. The post card also gains the same
three-dot menu affordance or the navbar share action is internally unified with
the same share helper. The detail page must also expose `编辑` when the current
user owns the post.

### Publish Page

The page title and CTA text change by mode:

- Create mode: `发布动态` / `发布`
- Edit mode: `编辑动态` / `保存`

When loading an existing post in edit mode, the page shows a loading state
before the form is ready. Submitting while images are uploading remains blocked
exactly as in create mode.

## Error Handling

- Unauthenticated edit attempts: backend returns auth failure; frontend routes
  user to login using the existing circle error pattern.
- Non-author edit attempts: backend returns a permission error; frontend shows
  the server message and does not open an edit-only UI path again after reload.
- Missing post in edit mode: publish page shows load failure or missing-post
  state and does not submit.
- Clipboard copy failure: show a non-blocking failure toast.

## Testing

### Frontend

- Unit tests for share-text generation helper.
- Unit tests for menu item resolution based on current user and post owner.
- Service test for `updateCirclePost`.
- If publish mode logic is extracted to helpers, add tests for create vs edit
  mode state derivation.

### Backend

- Request-level tests or focused controller/service tests for:
  - author can edit own post
  - non-author cannot edit
  - missing post returns failure

## Compatibility

- Existing create-post flow remains unchanged when no `id` route param exists.
- Existing post detail route remains the share target.
- The action menu must not break tap propagation for likes/comments/favorites.
- The ownership check must not rely on author display name, only stable user id.

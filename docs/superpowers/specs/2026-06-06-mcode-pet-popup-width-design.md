# MCode Pet Popup Width Design

**Date:** 2026-06-06  
**Scope:** `mcode-app` 移动端宠物相关弹层宽度与留白统一  
**Primary Files:** `mcode-app/src/components/pet/PetFloat.vue`, `mcode-app/src/components/pet/PetPanel.vue`

---

## Problem

当前宠物相关的两个弹层在移动端横向占用过大，视觉上贴边，容易产生“遮挡内容”和“压屏”的感觉：

- 首次选择宠物弹窗位于 [`PetFloat.vue`](D:/Repos/xyito/lingyun/mcode/mcode-app/src/components/pet/PetFloat.vue)，当前使用固定 `min-width: 580rpx`
- 宠物详情面板位于 [`PetPanel.vue`](D:/Repos/xyito/lingyun/mcode/mcode-app/src/components/pet/PetPanel.vue)，当前底部弹层内容基本按整宽铺开

这两个弹层的视觉语言也不一致：

- 首次选择弹窗是中间弹出
- 宠物详情面板是底部弹出
- 但两者都缺少明确的移动端横向安全留白规则

用户要求：

- 统一处理两处宠物弹层
- `mcode` 不考虑桌面端
- 统一后的左右留白采用 `32rpx`

## Goal

为移动端宠物相关弹层建立一套统一的横向布局规则，满足以下目标：

- 首次选择宠物弹窗不再显得过宽
- 宠物详情面板不再以整宽压满屏幕
- 两者统一使用 `32rpx` 左右留白
- 保持现有交互方式不变
- 只改布局表现，不改宠物业务逻辑

## Non-Goals

- 不调整宠物交互逻辑
- 不调整宠物成长、经验、皮肤、配饰逻辑
- 不改首次选择流程
- 不改详情面板的 tab 结构
- 不做桌面端适配
- 不顺带重做其他非宠物弹层

## Options Considered

### 1. 只缩内部内容，不改弹层容器

**做法**

- 缩小卡片项尺寸、padding 和文字
- 保持现有弹层宽度策略

**优点**

- 改动最小

**缺点**

- 不能真正解决“弹层太宽”
- 用户感知改善有限

### 2. 保持现有弹出方式，统一改成移动端窄卡片容器（采用）

**做法**

- 首次选择宠物弹窗继续使用 `mode="center"`
- 宠物详情面板继续使用 `mode="bottom"`
- 但两者内部主容器统一增加移动端横向留白
- 首次选择弹窗去掉固定最小宽度，改成基于视口计算的窄卡片
- 详情面板做成底部浮起的收窄卡片，而不是整宽铺开

**优点**

- 直接解决横向遮挡问题
- 视觉语言更统一
- 不影响现有交互和数据流

**缺点**

- 比单纯缩内容多一些容器样式调整

### 3. 改成完全居中的小弹窗体系

**做法**

- 首次选择弹窗和详情面板都改成居中卡片

**优点**

- 视觉上最统一

**缺点**

- 详情面板从底部抽屉改成中间弹窗会影响当前交互习惯
- 改动超出本次最小范围

## Chosen Design

采用 **方案 2**：

- 保留两处弹层原有的弹出方式
- 统一为移动端窄卡片视觉
- 统一使用 `32rpx` 左右留白

核心原则如下：

1. **只解决宽度和留白问题。**
2. **不改变弹层类型。**
3. **首次选择弹窗与详情面板共享同一套横向约束思路。**
4. **移动端优先，不为桌面场景增加额外分支。**

## Detailed Design

### 1. 首次选择宠物弹窗

位置：[`PetFloat.vue`](D:/Repos/xyito/lingyun/mcode/mcode-app/src/components/pet/PetFloat.vue)

当前问题：

- `.pet-setup` 使用固定 `min-width: 580rpx`
- 在窄屏设备上容易接近贴边
- 视觉上像“大块铺开”，而不是“可选择的小卡片”

调整方向：

- 去掉固定最小宽度
- 改为基于视口的宽度计算
- 两侧固定预留 `32rpx`

推荐规则：

- 主容器宽度遵循 `屏幕宽度 - 64rpx`
- 允许保留一个合理的内部 padding
- 保证横向滚动的物种列表仍可正常滑动

结果预期：

- 弹窗明显收窄
- 左右边界更轻
- 不会影响物种横向滚动和输入框可用性

### 2. 宠物详情面板

位置：[`PetPanel.vue`](D:/Repos/xyito/lingyun/mcode/mcode-app/src/components/pet/PetPanel.vue)

当前问题：

- 底部弹层虽然有圆角，但内容容器仍接近整宽
- 视觉上压迫感较强
- 与首次选择弹窗缺少统一的横向节奏

调整方向：

- 保留 `mode="bottom"`
- 让 `.pet-panel` 不再直接铺满可视宽度
- 通过左右外边距或等效宽度约束，形成一个“底部浮起卡片”

推荐规则：

- 面板内容统一遵循 `32rpx` 左右留白
- 面板保持现有垂直结构和滚动区域
- 圆角、底部关闭把手、tab、内容滚动逻辑不变

结果预期：

- 面板仍然从底部出现
- 但视觉上更像浮起卡片，而不是整屏底板
- 与首次选择弹窗形成一致的横向密度

### 3. 统一布局规则

本次为宠物相关弹层定义一套简单的一致性规则：

- 仅作用于移动端
- 横向安全留白统一为 `32rpx`
- 不依赖桌面断点
- 不额外区分平板和手机

一致性体现在：

- 左右边缘距离一致
- 容器不再使用固定的大最小宽度
- 弹层主体更靠近“卡片”而不是“整幅面板”

### 4. 不修改的部分

以下内容保持不变：

- `up-popup` 的显示与关闭逻辑
- 首次选宠物流程
- 宠物详情 tab 内容结构
- 滚动容器层级
- 宠物渲染与状态逻辑

## Data Flow

本次是纯样式和布局调整，不改变数据流。

弹层打开、关闭、状态切换、宠物初始化、详情面板展示等逻辑均保持现状。

## Implementation Notes

推荐只通过容器样式完成，不新增额外状态。

建议实现方式：

- 在 [`PetFloat.vue`](D:/Repos/xyito/lingyun/mcode/mcode-app/src/components/pet/PetFloat.vue) 中调整 `.pet-setup`
- 在 [`PetPanel.vue`](D:/Repos/xyito/lingyun/mcode/mcode-app/src/components/pet/PetPanel.vue) 中调整 `.pet-panel`

重点避免：

- 再次引入固定的大 `min-width`
- 为桌面端增加复杂分支
- 通过压缩内部内容来替代真正的宽度约束

## Error Handling

本次不涉及运行时错误处理。

需要关注的风险主要是样式层面：

- 物种卡片横向滚动是否仍顺畅
- 输入框是否在窄化后显得过挤
- 详情面板内容区是否因收窄导致 tab 或成就列表换行异常

这些都应通过手工验证确认。

## Testing Strategy

手工验证以下场景：

1. 首次打开且未初始化宠物
   - 预期：首次选择宠物弹窗横向收窄
   - 预期：左右各保留约 `32rpx` 视觉留白

2. 首次选择弹窗中的物种横向滑动
   - 预期：滚动正常
   - 预期：选中态样式不受影响

3. 首次选择弹窗中的输入框与确认按钮
   - 预期：仍可正常输入和点击

4. 点击悬浮宠物打开详情面板
   - 预期：仍从底部弹出
   - 预期：面板视觉宽度收窄，不再整宽压屏

5. 详情面板切换 tab
   - 预期：状态、皮肤、成就三个 tab 展示正常

6. 详情面板内容滚动
   - 预期：滚动区域和关闭把手不受影响

## Acceptance Criteria

- 首次选择宠物弹窗不再使用当前固定宽感
- 宠物详情面板不再以接近整宽的方式压满屏幕
- 两处弹层统一遵循 `32rpx` 左右留白
- 两处弹层的交互方式保持不变
- 宠物初始化、详情切换、tab 切换、滚动等功能不回归

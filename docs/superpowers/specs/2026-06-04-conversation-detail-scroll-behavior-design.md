# Conversation Detail Scroll Behavior Design

**Date:** 2026-06-04  
**Scope:** `mcode-app` 会话详情页滚动、自动跟随、输入区高度联动  
**Primary File:** `mcode-app/src/pages/conversation-detail/index.vue`

---

## 背景

当前会话详情页已经具备以下能力：

- 首屏加载后主动调用 `scrollToBottom(true)`。
- 用户上滑时通过 `shouldAutoFollowBottom` 控制是否继续自动跟随。
- 输入区使用固定定位，消息列表高度通过 `measureMessageListHeight()` 动态计算。

但现有实现仍有三个用户可见问题：

1. **进入详情页后滚动策略不稳定**
   - 首次进入会滚到底部。
   - 返回同一会话时虽然已有 `scrollAnchor` 恢复逻辑，但会被后续无条件 `scrollToBottom(true)` 覆盖。

2. **实时消息更新时不会稳定自动滚动**
   - 当前自动跟随主要依赖 `watch(messages.value.length)`。
   - 流式回复通常只更新同一条 `liveMessage` 内容，消息条数不变，因此不会触发滚动。

3. **输入框增高时消息区域高度没有稳定联动**
   - 输入区高度变化主要来自 `textarea autoHeight`、附件区、待发送队列、slash 面板、键盘高度。
   - 当前只在首屏和历史状态文案变化时重测高度，导致输入区变高时底部消息可能被遮挡。

---

## 目标

- 首次进入或没有可恢复位置时，消息列表直接定位到底部。
- 返回同一会话时，优先恢复用户上次阅读位置，而不是强制到底部。
- 当用户当前接近底部时，实时消息更新自动跟随到底部。
- 当用户已经离开底部查看历史消息时，实时更新不抢占当前阅读位置。
- 输入区任何会导致高度变化的交互都应同步驱动消息列表高度重算，避免底部消息被遮挡。

---

## 非目标

- 不改会话详情页整体布局形态，仍保留 `scroll-view` 作为消息滚动容器。
- 不引入新的数据库迁移；本次滚动位置恢复仅依赖页面级缓存。
- 不调整会话历史加载策略、消息渲染结构或 ACP 事件协议。

---

## 方案对比

### 方案 A：继续沿用 index 锚点，仅补充几个 watch

**做法**
- 保持 `msg-${index}` 作为滚动锚点。
- 追加对输入区高度和流式内容的监听。

**问题**
- 历史消息前插后，index 会整体偏移，锚点不稳定。
- 返回同一会话时仍然难以准确恢复用户阅读位置。

### 方案 B：只记录 `scrollTop`

**做法**
- 用户滚动时记录当前 `scrollTop`。
- 返回详情页后直接恢复 `scrollTop`。

**问题**
- 历史 prepend、消息高度变化、布局高度变化后，单独的 `scrollTop` 容易漂移。
- 在“前插历史后保持当前可视内容”场景下不够稳。

### 方案 C：稳定消息锚点 + `scrollTop` + `nearBottom` 状态分离（采用）

**做法**
- 使用稳定消息 id 作为程序化定位锚点。
- 使用 `scrollTop` 和 `nearBottom` 表示用户真实阅读位置。
- 进入、实时更新、输入区高度变化统一走同一套视口同步规则。

**优点**
- 首次进入、返回恢复、实时自动跟随、输入区增高四类场景可以统一处理。
- 与现有 `scroll-view` 结构兼容，改动集中在详情页与缓存结构。

---

## 决策

采用 **方案 C**。

核心原则如下：

1. **程序化定位** 与 **用户阅读位置** 分离。
2. **滚到底部** 只在明确需要时触发，不再作为通用收尾动作。
3. **自动跟随** 只在用户接近底部时启用。
4. **输入区高度变化** 和 **实时消息变化** 使用统一的视口同步入口。

---

## 详细设计

### 1. 稳定消息锚点

当前消息节点使用 `msg-${index}` 作为 id。该方案在历史消息前插时不稳定，因为所有后续消息的 index 都会变化。

改为：

- 消息节点 id 使用稳定消息标识派生，例如 `msg-${msg.id}`。
- `scrollIntoView` 仅用于以下场景：
  - 首次进入时跳到底部。
  - 上拉加载历史后，重新回到插入前第一条可见消息。
  - 页面恢复时基于锚点兜底恢复。

这样即便前面插入新消息，目标节点仍然可被准确定位。

### 2. 视图状态拆分

当前实现把 `scrollIntoView` 同时当作“程序化滚动目标”和“用户阅读位置缓存”，会导致语义混乱。

改为维护两类状态：

- **程序化滚动状态**
  - `scrollIntoView`
  - 仅在主动触发跳转时写入。

- **用户阅读状态**
  - `scrollTop`
  - `nearBottom`
  - `anchorMessageId`

其中：

- `scrollTop` 表示滚动容器当前像素位置。
- `nearBottom` 表示用户是否处于底部跟随区间，继续沿用现有阈值语义。
- `anchorMessageId` 表示当前阅读位置对应的稳定消息锚点，用于页面恢复和历史 prepend 场景。

### 3. 页面进入规则

进入会话详情页时，按以下优先级处理：

1. **存在缓存且上次不在底部**
   - 优先恢复 `scrollTop`。
   - 若 `scrollTop` 不可用但 `anchorMessageId` 存在，则使用锚点恢复。

2. **存在缓存且上次在底部附近**
   - 直接跳到底部。

3. **无有效缓存**
   - 视为首次进入，直接跳到底部。

同时增加一个临时恢复标记，例如 `isRestoringScroll`：

- 页面正在执行恢复时，抑制自动跟随逻辑。
- 恢复完成后，再按 `nearBottom` 规则决定后续实时更新是否跟随。

这样可以避免“刚恢复到历史位置，流式消息一到又被强制拉到底部”。

### 4. 实时更新自动跟随

当前只监听 `messages.length` 不足以覆盖流式消息，因为流式更新通常修改现有 `liveMessage` 内容。

改为监听“当前渲染消息内容的变化”，覆盖以下触发源：

- 新增乐观用户消息。
- `liveMessage` 文本 delta。
- tool call / tool call update。
- `turn_complete` 后的最终落盘替换。

自动跟随规则：

- `nearBottom === true`：执行滚到底部。
- `nearBottom === false`：保持当前阅读位置，不滚动。

额外规则：

- 用户主动发送消息时，无论当前是否在底部，都应滚到底部，因为用户已明确切回当前对话流。

### 5. 输入区高度联动

输入区高度变化可能由以下因素触发：

- `up-textarea` 自动增高。
- 键盘高度变化。
- 附件预览区出现/消失。
- 待发送队列展开/收起。
- slash 面板出现/消失。
- 历史状态条出现/消失。

引入统一入口，例如 `scheduleViewportSync(reason)`，负责：

1. `nextTick` 等待布局稳定。
2. 调用 `measureMessageListHeight()` 重算消息列表高度。
3. 根据当前状态决定：
   - `nearBottom === true`：重算后滚到底部。
   - `nearBottom === false`：保留当前 `scrollTop`，不改变阅读位置。

`up-textarea` 直接接入以下事件：

- `linechange`
- `keyboardheightchange`

同时对附件、队列、slash 面板、历史状态条的显隐变化补充相同同步入口。

### 6. 消息列表高度计算

当前高度计算虽然已经拿到了：

- `toolbarBottom`
- `historyStatusBottom`
- `inputTop`

但最终使用的仍然是“窗口高度减各块高度”的近似值。

改为使用真实几何关系：

- `headerBottom = historyStatusBottom || toolbarBottom`
- `messageViewportHeight = max(0, inputTop - headerBottom)`

这样消息区域会直接跟随顶部状态区和底部输入区的真实位置变化，不依赖近似减法。

### 7. 上拉历史加载的稳定性

历史消息前插时，仍保持“插入前第一条可见消息位置不变”的体验。

具体规则：

- 读取插入前第一条可见消息的稳定消息 id。
- prepend 完成后，使用该稳定锚点恢复到同一消息。

不再依赖 index 锚点，因此历史加载后不会因数组前插导致锚点偏移。

### 8. 缓存策略

本次仅更新页面级缓存结构 `conversationCache`：

- 新增 `scrollTop`
- 新增 `nearBottom`
- 新增 `anchorMessageId`

离开页面时持久化这些字段；重新进入时优先使用它们恢复阅读位置。

本次不扩展 `conversation_runtime` 数据表，不为该需求增加 SQLite 字段与迁移。

这样可以先解决页面往返场景；若未来需要跨进程恢复，再单独评估是否下沉到数据库。

## 数据流

### 首次进入 / 无缓存

1. 加载会话详情与本地消息。
2. 完成首轮布局测量。
3. 将消息列表定位到底部。
4. 标记进入后的自动跟随状态为 `nearBottom = true`。

### 返回进入 / 有缓存

1. 加载会话详情与本地消息。
2. 完成首轮布局测量。
3. 若缓存表明上次不在底部：
   - 优先恢复 `scrollTop`。
   - 失败时降级为 `anchorMessageId`。
4. 恢复期间禁止自动跟随。
5. 恢复结束后重新开启实时跟随判定。

### 实时消息流入

1. ACP 事件更新会话 runtime。
2. 页面监听到渲染消息内容变化。
3. 进入统一视口同步入口。
4. 若 `nearBottom = true` 则到底部，否则保位。

### 输入区增高

1. `textarea` 行数变化或键盘高度变化触发事件。
2. 附件区、队列区、slash 面板显隐变化触发监听。
3. 统一进入视口同步入口。
4. 重算消息列表高度并按当前 `nearBottom` 状态处理滚动。

---

## 错误处理与降级

- 若布局测量返回 0 或异常值，不立即覆盖现有消息区高度，保留上一轮有效高度并在下一次视图更新后重试。
- 若恢复时缓存的 `anchorMessageId` 对应节点不存在，降级为恢复 `scrollTop`；若 `scrollTop` 也不可用，则降级到底部。
- 若实时更新发生在恢复过程中，先仅更新数据，不触发到底部，待恢复完成后再由下一次同步入口接管。
- 若历史 prepend 后找不到插入前锚点，保持当前滚动状态，不额外触发滚动跳变。

## 验证方案

### 手工验证

1. **首次进入**
   - 打开一个已有消息的会话。
   - 预期直接看到最后一条消息。

2. **返回恢复**
   - 在同一会话中上滑到历史位置。
   - 返回列表再重新进入。
   - 预期恢复到上次阅读位置，而不是跳到最底部。

3. **实时自动跟随**
   - 保持页面在底部附近，触发流式回复。
   - 预期回复增长过程中持续跟随到底部。

4. **历史阅读不被打断**
   - 上滑离开底部，再触发流式回复。
   - 预期新消息到来时列表不自动抢到最底部。

5. **输入框多行**
   - 输入多行文本，触发 `textarea` 增高。
   - 预期消息列表高度同步缩短，底部消息不被输入区遮挡。

6. **附件 / 队列 / slash 面板**
   - 打开附件预览、展开待发送队列、显示 slash 面板。
   - 预期消息列表高度同步变化，无遮挡、无明显跳变。

7. **历史加载**
   - 上拉触发加载更早消息。
   - 预期 prepend 后当前阅读位置不跳。

### 回归重点

- 不影响现有历史加载逻辑。
- 不影响发送消息、乐观消息、turn 完成后的消息渲染。
- 不影响工具调用消息和计划任务浮层展示。

---

## 实施边界

本设计只解决会话详情页的滚动行为一致性问题。

不包含：

- 跨应用重启的滚动位置恢复。
- 会话列表页、概览页、其他详情页的统一滚动抽象。
- 消息列表虚拟化或性能优化。

当前范围足以覆盖用户提出的三个问题，并且不引入额外数据库迁移风险。

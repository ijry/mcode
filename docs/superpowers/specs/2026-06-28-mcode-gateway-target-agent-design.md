# MCode Gateway Target Agent Design

## Background

The original P1 connection model separates the final agent from the route:

- `targetAgent = codeg | opencode | mcode-desktop`
- `routeMode = direct | gateway`
- `gatewayProvider = official | custom`

The current add-connection implementation incorrectly narrows `routeMode = gateway`
to `targetAgent = mcode-desktop`. That blocks a real deployment where the same
computer exposes Codeg, OpenCode, and MCode Desktop through the same gateway.

This design corrects that drift. `routeMode` answers "how does MCode reach the
agent". `targetAgent` answers "which agent is behind the connection". They must
remain independent in both direct and gateway modes.

## Goals

- Let gateway-mode connections select `Codeg`, `OpenCode`, or `MCode Desktop`.
- Preserve the existing relay session model where a token is bound to one
  logical `targetId`.
- Support one physical computer registering multiple logical gateway targets,
  such as `codeg@pc`, `opencode@pc`, and `mcode-desktop@pc`.
- Keep official Codex CLI and Claude CLI behind `targetAgent = mcode-desktop`
  capabilities, not as mobile-side target agents.
- Prevent accidental pairing against the wrong target type by validating pair
  metadata in the app.

## Non-Goals

- Do not make MCode Desktop the mandatory host for Codeg or OpenCode.
- Do not add mobile-side `targetAgent = codex` or `targetAgent = claude`.
- Do not change relay into a business-aware router for Codeg/OpenCode/Desktop
  payloads; relay continues to forward by `targetId`.

## Chosen Approach

Gateway mode uses the same target selector as direct mode:

- Direct targets: `Codeg`, `OpenCode`, `MCode Desktop`
- Gateway targets: `Codeg`, `OpenCode`, `MCode Desktop`

For gateway mode, the user selects:

- target agent
- gateway provider
- optional custom gateway domain
- pair code and pair secret

After `POST /v1/pair`, the app compares the returned target metadata against
the selected `targetAgent`. If the pair code belongs to a different target type,
the app rejects the connection with a clear error and does not save it.

The relay remains mostly unchanged because it already stores `targetAgent` on
`PairOfferRecord` and `TargetRecord`, and sessions already bind to `targetId`.
The key requirement is that upstream agents register independent logical
targets when they want to be separately selectable through the gateway.

## Data Flow

Gateway pairing flow:

1. An upstream agent registers or offers a pair code with:
   - `targetId`
   - `targetAgent`
   - `targetName`
   - `capabilities`
   - `protocolVersion`
2. MCode app creates a gateway connection form with the user-selected
   `targetAgent`.
3. MCode app calls relay `POST /v1/pair`.
4. Relay returns session tokens plus target metadata.
5. MCode app validates:
   - returned target metadata exists when available
   - returned `targetAgent` equals the selected `targetAgent`
6. MCode app saves:
   - `routeMode = gateway`
   - selected and verified `targetAgent`
   - gateway provider and base URL
   - pair code and secret
   - gateway session
   - target profile metadata

If the relay is an older deployment and does not return `targetAgent`, the app
may keep legacy compatibility for existing records, but new manual gateway
connections should prefer explicit metadata and surface a compatibility warning
or failure rather than silently changing the selected target.

## UI Behavior

The add-connection form shows the target selector for both route modes.

Direct mode fields:

- target agent selector
- service address
- access token

Gateway mode fields:

- target agent selector
- gateway provider selector
- custom domain input when provider is `custom`
- pair code
- pair secret

The old gateway hint saying the gateway "defaults to MCode Desktop" is removed.
New copy should say that the pair code must be generated by the selected target.

Connection cards continue to show both target type and route mode. For example:

- `Codeg / 网关`
- `OpenCode / 网关`
- `MCode Desktop / 网关`

## Compatibility

- Existing saved gateway connections remain valid.
- Existing legacy relay records still migrate to `targetAgent = codeg`.
- Existing MCode Desktop gateway connections keep using
  `targetAgent = mcode-desktop`.
- Pair metadata remains the source of truth for target display name,
  capabilities, protocol version, and target id.
- If a saved connection is edited, the form must preserve its saved
  `targetAgent` instead of forcing gateway mode to `mcode-desktop`.

## Native iOS/Android Replication Guidance

Native clients must implement the same rule: `routeMode` never determines
`targetAgent`.

For gateway-mode creation:

- show a target selector
- pair with the selected gateway
- compare pair response `targetAgent` with the selected target
- reject mismatches before saving
- persist `targetAgent` and `targetProfile`

For runtime routing:

- choose drivers by `targetAgent + routeMode`
- treat official CLI features as capabilities of `mcode-desktop`
- do not special-case gateway mode as Desktop-only

## Testing

Required app tests:

- gateway form target selector remains visible in gateway mode
- changing route mode to gateway does not overwrite `targetAgent`
- gateway save persists the selected `targetAgent`
- pair metadata mismatch rejects the connection
- editing a gateway OpenCode/Codeg connection preserves its target

Required relay tests:

- pair responses still include `targetAgent`
- `/v1/targets` can list multiple targets with different `targetAgent` values
  from the same deployment
